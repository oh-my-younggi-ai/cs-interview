# 한 줄 정의

1차 캐시는 [영속성 컨텍스트](/jpa/persistence-context.md) 내부에서 엔티티를 식별자(`@Id`) 기준으로 보관하는 캐시로, 같은 트랜잭션 안에서 동일 엔티티를 다시 조회할 때 DB를 거치지 않고 즉시 반환한다.

# 핵심 개념

- 엔티티가 영속 상태가 되면 `Map<식별자, 엔티티>` 형태로 1차 캐시에 보관된다.
- `find()` 호출 시: **1차 캐시 먼저 확인 → 있으면 반환, 없으면 DB 조회 후 캐시에 저장**.
- **동일성(identity) 보장**: 같은 트랜잭션에서 같은 식별자로 두 번 조회하면 같은 인스턴스(`a == b` true)를 반환한다.
- 범위는 **트랜잭션(영속성 컨텍스트) 단위**로 매우 짧고, 트랜잭션이 끝나면 사라진다.

```
em.find(Member.class, 1L);  // 1차 캐시 없음 → DB 조회, 캐시에 저장
em.find(Member.class, 1L);  // 1차 캐시 히트 → DB 안 감, 같은 인스턴스 반환
```

# 면접 단골 질문

- Q: 1차 캐시란 무엇인가요?
  - 포인트: 영속성 컨텍스트 내부 캐시, 식별자 기반, 트랜잭션 범위, 동일성 보장.
- Q: 1차 캐시와 2차 캐시의 차이는?
  - 포인트: 1차는 트랜잭션 범위(짧음, 기본 활성), 2차는 애플리케이션/여러 트랜잭션 공유(별도 설정 필요).
- Q: 1차 캐시 덕분에 보장되는 것은?
  - 포인트: 반복 조회 시 DB 접근 절감 + 동일성(==) 보장.

# 헷갈리는 점 / 함정

- 1차 캐시는 성능 최적화보다는 **동일성 보장과 영속성 컨텍스트 동작의 기반**이라는 의미가 더 크다 (트랜잭션이 짧아 캐시 적중 효과는 제한적).
- JPQL로 쿼리하면 1차 캐시를 먼저 보지 않고 **DB에 직접 쿼리**한다 (단, 결과를 영속화할 때 이미 캐시에 있으면 캐시본 사용).
- 애플리케이션 전역 캐시(2차 캐시)와 혼동 금지.
