# 한 줄 정의

**프로세스**는 독립적인 메모리 공간을 가진 실행 단위이고, **스레드**는 프로세스 안에서 메모리를 공유하는 더 가벼운 실행 흐름이다.

# 핵심 개념

## 프로세스 (Process)

- OS가 자원(메모리, 파일 핸들, 소켓 등)을 배정하는 **기본 단위**.
- 각 프로세스는 독립적인 **가상 주소 공간**을 가진다.
  - Code(Text), Data(전역변수), Heap(동적 할당), Stack(지역변수·호출 스택) 영역이 분리됨.
- 한 프로세스의 오류가 다른 프로세스에 영향을 주지 않는다 (격리).
- 프로세스 간 통신(IPC)은 파이프, 소켓, 공유 메모리 등 별도 메커니즘 필요.

## 스레드 (Thread)

- 프로세스 안에서 **CPU 스케줄링을 받는 실행 흐름**.
- 같은 프로세스의 스레드들은 **Code, Data, Heap 영역을 공유**한다.
- 각 스레드는 **독립적인 Stack과 레지스터(PC 포함)** 를 가진다.
- 스레드 간 통신은 공유 메모리를 그대로 쓰므로 빠르지만, 동기화가 필요하다.

## 핵심 비교표

| 항목 | 프로세스 | 스레드 |
|------|---------|--------|
| 메모리 공간 | 독립 (가상 주소 공간 분리) | 공유 (Heap·Data·Code 공유) |
| 생성 비용 | 높음 (fork 시 주소 공간 복사) | 낮음 |
| 컨텍스트 스위치 비용 | 높음 (TLB 플러시, 주소 공간 교체) | 낮음 (같은 주소 공간 유지) |
| 격리성 | 강함 (크래시가 다른 프로세스에 전파 안 됨) | 약함 (한 스레드 오류가 프로세스 전체에 영향) |
| 통신 | IPC (파이프, 소켓, 공유 메모리 등) | 공유 메모리 직접 접근 |
| 동기화 필요성 | 낮음 | 높음 (Race Condition 주의) |
| 예시 | 크롬 탭마다 별도 프로세스 | 한 서버 프로세스 안의 요청 처리 스레드들 |

## 메모리 구조 (시각화)

```
프로세스 A                  프로세스 B
┌───────────────┐           ┌───────────────┐
│ Code          │           │ Code          │
│ Data          │           │ Data          │
│ Heap          │           │ Heap          │
│ Stack(T1)─┐  │           │ Stack(T1)─┐  │
│ Stack(T2)─┤  │           └───────────┴──┘
└───────────┴──┘
   T1, T2는 Code/Data/Heap 공유
```

# 면접 단골 질문

- Q: 프로세스와 스레드의 차이를 설명해보세요.
  - 포인트: 메모리 공간의 독립 vs 공유, 생성·전환 비용 차이, 격리성 vs 효율성 트레이드오프.

- Q: 스레드를 쓰면 왜 빠른가요?
  - 포인트: 컨텍스트 스위치 시 주소 공간 교체 불필요(TLB 미플러시), 생성 비용 낮음, IPC 없이 공유 메모리 직접 접근.

- Q: 멀티프로세스 vs 멀티스레드, 어떤 걸 선택하나요?
  - 포인트: 안정성이 중요하면 멀티프로세스(크롬, nginx의 worker 모델), 성능·메모리 효율이 중요하면 멀티스레드. 파이썬 GIL 같은 언어 제약도 언급.

- Q: 컨텍스트 스위치가 무엇이고, 프로세스와 스레드에서 비용이 왜 다른가요?
  - 포인트: CPU 레지스터·PC 저장/복원. 프로세스는 추가로 MMU·TLB·페이지 테이블 교체 필요.

- Q: 스레드가 메모리를 공유하면 어떤 문제가 생기나요?
  - 포인트: Race Condition, Deadlock. 뮤텍스/세마포어/모니터 등의 동기화 기법 필요.

# 헷갈리는 점 / 함정

- **"스레드는 Stack도 공유한다"** — 오답. 각 스레드는 독립적인 Stack을 가진다. 공유하는 건 Code·Data·Heap.
- **"멀티스레드가 항상 빠르다"** — 오답. 동기화 오버헤드(Lock 경합)가 병목이 될 수 있고, CPU 바운드 작업에서는 GIL처럼 언어 런타임 제약이 있을 수 있다.
- **"프로세스 생성은 무조건 메모리를 복사한다"** — 세부 오답. Linux `fork()`는 Copy-on-Write(CoW)를 사용하므로 실제 쓰기가 발생하기 전까지는 물리 페이지를 복사하지 않는다.
- **컨텍스트 스위치 비용 과소평가** — 프로세스 전환은 TLB 플러시로 캐시 미스가 급증해 성능 영향이 크다. 스레드 전환은 이 비용이 없다.
