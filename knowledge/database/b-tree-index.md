# 한 줄 정의

B-Tree 인덱스는 균형 트리로 키를 정렬 보관해 범위/정렬 검색을 빠르게 한다.

# 핵심 개념

- **균형(Balanced) 트리**: 루트→리프 높이가 모든 경로에서 같게 유지된다 → 어떤 키든 **O(log N)** 접근.
- **노드 = 디스크 페이지**: 한 노드에 키를 수십~수백 개 담아 트리 높이를 낮춘다(수억 건도 높이 3~4). 디스크 I/O 횟수를 최소화하는 설계.
- **정렬 보관**: 키가 정렬돼 있어 `=` 조회뿐 아니라 **범위(`BETWEEN`, `<`, `>`)·정렬(`ORDER BY`)·전방일치(`LIKE 'abc%'`)** 를 인덱스로 처리한다.
- **B-Tree vs B+Tree**: 실무 DB(MySQL InnoDB 등)는 대부분 **B+Tree** — 데이터(또는 포인터)를 리프에만 두고 **리프끼리 연결 리스트**로 이어 범위 스캔이 더 빠르다.

| 항목 | B-Tree 인덱스 | 해시 인덱스 |
|------|---------------|-------------|
| 동등 조회(=) | O(log N) | O(1)에 가까움 |
| 범위/정렬/전방일치 | **가능** (정렬 보관) | 불가 |
| 대표 용례 | 범용 (대부분의 DB 기본) | 메모리 테이블, 일부 특수 용도 |

## 인덱스를 타지 못하는 경우 (자주 묻는 포인트)

- 인덱스 컬럼을 **가공/함수 적용** (`WHERE UPPER(name) = ...`)
- **좌측 불일치** LIKE (`LIKE '%abc'`)
- 복합 인덱스에서 **선두 컬럼 없이** 뒤 컬럼만 조건에 사용

# 면접 단골 질문

- Q: B-Tree 인덱스가 범위 검색에 유리한 이유는?
  - 포인트: 정렬된 리프, 균형 높이. (B+Tree라면 리프 연결 리스트로 순차 스캔.)
- Q: 인덱스를 걸면 항상 빨라지나요?
  - 포인트: 조회는 빨라져도 **쓰기(INSERT/UPDATE/DELETE)마다 인덱스 갱신 비용**. 카디널리티 낮은 컬럼은 효과 미미.
- Q: 해시 인덱스 대신 B-Tree가 기본인 이유는?
  - 포인트: 동등 조회만 되는 해시와 달리 범위·정렬·전방일치까지 커버하는 범용성.

# 헷갈리는 점 / 함정

- **"B-Tree = B+Tree"가 아니다.** 실무 DB 기본 인덱스는 대부분 B+Tree(리프에만 데이터, 리프 연결). 면접에서 구분해 말하면 가점.
- **"인덱스는 많을수록 좋다"는 오해** — 쓰기 비용·스토리지·옵티마이저 혼란 증가. 조회 패턴 기반으로 선별.
- 인덱스가 있어도 컬럼 가공·좌측 불일치 LIKE·낮은 선택도면 **옵티마이저가 풀스캔을 택할 수 있다.**
