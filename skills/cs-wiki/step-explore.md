# Step Explore: 대화 탐색 · 추천 정리

대상 개념이 명확하지 않을 때의 진입. **대화로 함께 찾거나(discuss)**, **무엇을 정리할지
추천(recommend)** 한 뒤, 거기서 나온 개념 중 **위키에 없는 것만** `step-ingest` 절차로 넘긴다.

## RULES

- **한 번에 한 질문/제안하고 사용자 답을 기다린다.** 사용자 답변을 대신 만들거나 추측해 이어가지 않는다.
- **위키에 이미 있는 개념은 다시 정리하지 않는다** — 루트/주제 `index.md`와 대조한다(중복 방지).
- 위키 파일은 **사용자 승인 전까지 만들지 않는다.** 실제 작성은 전부 `./step-ingest.md` 절차로.
- `{explore_mode}`(`discuss` | `recommend`)는 `step-01-route`에서 설정돼 있어야 한다.


## A. recommend — 무엇을 정리할지 모를 때

### 1. 위키 현황 파악
루트 `index.md`(필요시 주제 `index.md`)를 읽어 **이미 있는 개념 목록**을 파악한다.

### 2. 갭 기반 추천
흔한 CS 면접 주제(`network`, `operating-systems`, `database`, `data-structure`, `algorithm`,
`jpa`, `design-pattern`, `web`, `security`, `system-design` 등)에서 **위키에 아직 없는 고빈출 개념**을
2~5개 고른다. **이미 위키에 있는 개념은 제외한다.** 각 추천에 "왜 정리하면 좋은지" 한 줄 이유를 붙인다.

> 📚 위키에 아직 없는 것 중, 정리하면 좋은 주제:
> - **{개념}** — {한 줄 이유}
> - ...
> 어떤 걸 정리할까요? (또는 "얘기하며 정리"도 가능)

### 3. 선택 → 분기
사용자가 개념을 고르면 방식을 확인해 분기한다:
- **"바로 정리"** → `{target_concept}` 설정 후 `./step-ingest.md`.
- **"얘기하며"** → 아래 **B. discuss**로 진행.


## B. discuss — 대화하며 정리

### 1. 대화
사용자가 원하는 내용/질문에 대해 대화한다. **한 걸음씩** 설명·되묻기·예시로 개념이 자연스럽게
드러나게 하되, **매 턴 사용자 입력을 기다린다.** 답변을 대신 쓰지 않는다.

### 2. 키워드 추출
대화가 한 매듭 지어지면, 오간 내용에서 **CS 개념 키워드**를 뽑아 `{surfaced_concepts}`에 담는다.

### 3. 위키 대조 (diff)
각 키워드를 루트/주제 `index.md`와 대조해 분류한다:
- **위키에 있음** → 정리 대상에서 제외(원하면 `./step-query.md`로 보여줄 수 있음).
- **위키에 없음** → 정리 후보.

### 4. 제안 → 승인
정리 후보를 목록으로 제시하고(각 한 줄 이유) **어떤 걸 위키에 넣을지 승인**받는다.

> 대화에서 나온 개념 중 위키에 **없는** 것:
> - **{개념}** — {왜 정리 가치 있는지}
> - ...
> 이 중 어떤 걸 정리할까요?

### 5. 정리
승인된 개념을 하나씩 `./step-ingest.md` 절차로 정리한다(면접 관점 포함, 교차링크·index·log 갱신,
자체 점검). 여러 개면 순서대로.

작업 완료. 추가 요청이 있으면 `./step-01-route.md`로 다시 분류한다.
